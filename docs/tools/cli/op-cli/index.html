OP CLI Commands

This reference lists the commands that you can use to build plugin with OP CLI.

note

OP CLI is a dependency of the plugin project.

Developer need to use npx to execute command like: npx op [command] [arguments] [options]

Overviewâ€‹

CommandDescription
initInitialize project settings.
addAdd a ability or module for the project.
packupPack up the plugin project and build .opk file in the root directory, provide version control
loginStore the parameters used to obtain user credentials in a specific environment into config/local.yaml.
ciStore the parameters used to obtain user credentials in a specific branch for (gitlab) CI into config/ci-deploy.yaml.
pickteamFetch the team list and update the config file with the team information from the team list for local debugging or CI deployment.
invokeStart the plugin project locally and invoke one or several life-cycles of the plugin.
updateUpdate project content.
upgradeUpgrade @ones/cli-plugin and relative dependencies to lastest version which range of current major version.content.

initâ€‹

Initialize project settings.

It will be automatic execute after install npm dependencies, developer can check scripts fields on packages.json to learn more detail.

npx op init

addâ€‹

Add a ability or module for the project.

npx op add <target-type>

Examplesâ€‹

  1. Add module or sub-module for project
npx op add module
npx op add sub-module
  1. Add ability or sub-module for project
npx op add ability

Argumentsâ€‹

ArgumentDescription
target-typeSpecify a content type that you want to add, support types:
  • module
  • sub-module
  • ability

packupâ€‹

Pack up the plugin project and build .opk file in the root directory, provide version control.

npx op packup [filename]

Argumentsâ€‹

ArgumentDescription
filenameSpecify the name of the .opk file, default is the project name.

Optionsâ€‹

ArgumentRequirementsDescription
--bump <mode>@ones/cli-plugin v1.14.0+packup plug-in with bump options, no-modify: do not bump version, major: bump major version, minor: bump minor version, patch: bump patch version

loginâ€‹

Store the parameters used to obtain user credentials in a specific environment into config/local.yaml.

Argument baseURL and hostURL is protocol sensitive, different protocol type will be treated as a different scope.

After login successfully, CLI will store the scope parameters into ~/.ones/cli-plugin.yaml for local debugging.

And at the same time, this command will try to run npx op pickteam local automatically if the current project has never had a team picked before.

npx op login [options] [baseURL] [hostURL]

Argumentsâ€‹

ArgumentDescription
baseURLSpecify the environment url for login.
hostURLSpecify the host url for local debugging.

Optionsâ€‹

OptionAliasDescription
--username <username>-u <username>Username can be email or phone number.
--password <password>-p <password>Password.
--scope [url]-s [url]Use specific scope parameters login, default is current environment url, this option is mutually exclusive with -u, -p options.

About --scope optionâ€‹

note

In different operating systems, the path of credential storage will be different.

  • Linux/macOS: ~/.ones/cli-plugin.yaml
  • Windows: C:\Users\YOUR_USERNAME\.ones\cli-plugin.yaml
Content like this
scopes:
  https://foo.example.com:
    baseURL: https://foo.example.com
    username: foo
    password: fooPass
  https://bar.example.com:
    baseURL: https://bar.example.com
    username: bar
    password: barPass

It should be noted that when executing the login command and using the scope option:

  • This option is mutually exclusive with -u, -p options.
  • If the selected scope credential is invalid, enter the interactive question and answer to enter the new user credential information (the effect is equivalent to executing npx op login).

Examplesâ€‹

  1. Enter the interactive Q&A process and input the login parameters.
npx op login
  1. Login to https://dev.ones.com environment. And then enter the inquiry session flow input rest parameters: hostURL, username, password.
npx op login https://dev.ones.com
  1. Login to https://dev.ones.com environment and set hostURL to tcp://dev.ones.com:9006 with two parameters and not enter the interactive Q&A process.
npx op login https://dev.ones.com tcp://dev.ones.com:9006 -u test@ones.com -p password
  1. Use https://dev.ones.com scope parameters to login to https://partnerdev.ones.com environment.
npx op login https://partnerdev.ones.com tcp://dev.ones.com:9006 -s https://dev.ones.com

ciâ€‹

Store the parameters used to obtain user credentials in a specific branch for (gitlab) CI into config/ci-deploy.yaml.

Argument url is protocol sensitive, different protocol type will be treated as a different scope.

Unlike the login command, current command does not store the scope parameters into ~/.ones/cli-plugin.yaml.

But if the current branch has never picked a team after the run is complete, npx op pickteam ci -b $currentSpecifyBranch will be automatically executed which is consistent.

npx op ci [options] [url]

Argumentsâ€‹

ArgumentDescription
urlSpecify the environment url for login.

Optionsâ€‹

OptionAliasDescription
--branch-name <branch-name>-b <branch-name>Specify the branch name for CI deployment configuration, default is master or main branch.
--username <username>-u <username>Username can be email or phone number.
--password <password>-p <password>Password.
--scope [url]-s [url]Use specific scope parameters login, default is current environment url, this option is mutually exclusive with -u, -p options.

Exampleâ€‹

  1. Enter the interactive Q&A process and input the parameters for CI deployment.
npx op ci
  1. Specify the environment address for the CI deployment as https://dev.ones.com, and then enter the interactive Q&A process input rest parameters: branch, username and password.
npx op ci https://dev.ones.com
  1. Specify the environment address for the CI deployment as https://dev.ones.com, and not enter the interactive Q&A process input rest parameters.
npx op ci https://dev.ones.com -b next -u test@ones.com -p password
  1. Use https://dev.ones.com scope parameters and specify the environment address for the CI deployment as https://partnerdev.ones.com.
npx op ci https://partnerdev.ones.com -b next -s https://dev.ones.com

pickteamâ€‹

Fetch the team list with the user credentials stored in the config file (config/local.yaml or config/ci-deploy.yaml) and update the config file with the team information from the team list for local debugging or CI deployment.

Before performing this operation, the login operation should be completed in the corresponding target.

npx op pickteam [target] [options]

Argumentsâ€‹

ArgumentDescription
targetSpecify the target type configuration for which team information needs to be updated, target type only can be local or ci.

Optionsâ€‹

OptionAliasDescription
--branch-name <branch>-b <branch>Specify a branch name for the CI deployment configuration.
--team-uuid <uuid>Match the specified team uuid in the team list.
--team-name <name>Match the specified team name in the team list.
note

The -b --branch-name <branch> option only takes effect when the target is ci, and leave blank or not specify a value will enter the interactive Q&A for branch name selection.

note

When --team-uuid and --team-name are specified at the same time, only --team-uuid will be used.

Exampleâ€‹

  1. Fetch the team list with the user credentials stored in the config/local.yaml. If does not store user credentials, you need to execute npx op login first.
npx op pickteam local
  1. Display the branch list with the user credentials stored in the config/ci-deploy.yaml, and enter the interactive Q&A process for choosing the branch to fetch the team list, if does not store user credentials, you need to execute npx op ci first.
npx op pickteam ci
  1. Fetch the team list with the user credentials stored in config/ci-deploy.yaml with the branch name next, please note that the branch name is case sensitive, and the branch name must be the same as the branch name in the config/ci-deploy.yaml, the CLI will throw an error if no branch name matches.
npx op pickteam ci -b next
  1. Fetch the team list with the user credentials stored in the config/local.yaml, after fetching the team list, CLI will try to pick the team with the team UUID T7YB134K, the CLI will throw an error if no team UUID matches
npx op pickteam local --team-uuid T7YB134K
  1. Fetch the team list with the user credentials stored in config/ci-deploy.yaml with the branch name next, after fetching the team list, CLI will try to pick the team with the team name TestTeamName, the CLI will throw an error if no team name matches
npx op pickteam ci -b next --team-name TestTeamName

invokeâ€‹

Start the plugin project locally and invoke one or several life-cycles of the plugin.

The difference between process and life-cycles is that the process will contain multiple life-cycles.

npx op invoke [options] <target>

Argumentsâ€‹

ArgumentDescription
targetSpecify the process or life-cycle you want to invoke.

Available life-cycleâ€‹

  • install
  • enable
  • start
  • stop
  • disable
  • uninstall

Developer can learn more about available life-cycle in the guide section.

Available processesâ€‹

ProcessContain life-cycle
run
  • install
  • enable
  • start
clear
  • stop
  • disable
  • uninstall

Optionsâ€‹

OptionAliasDescription
--mode <type>specify the mode, support mode:
  • all
  • backend
  • frontend
Default mode is all.
--webpack-stats-preset <level>Specify the webpack logging level, support level:
  • errors-only
  • errors-warnings
  • minimal
  • none
  • normal
  • verbose
  • detailed
  • summary
Default level is info.
